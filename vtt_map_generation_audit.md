# TALEWEAVERS: VTT Map Generation & Playability Audit

This document serves as an architectural audit of the TALEWEAVERS Virtual Tabletop (VTT) map generation logic, specifically detailing how the system creates visual player-level maps, where the data originates, how NPCs are scheduled, and an assessment of the current playability for playtesting.

## 1. The 4-Tier Hierarchical Mapping System

The VTT organizes the S.A.G.A. world into four distinct scales, managed by `useGameStore` and `useWorldStore` in the React frontend, and generated by the `TacticalGenerator` in the backend (`saga_director`).

*   **Tier 1: Global/Parchment View** (Macro Hexes)
    *   Displays the entire world map using hex cells. Rendered using simple circles or Delaunay triangulation/Voronoi cells for a parchment-like appearance.
*   **Tier 2: Regional View** (20x20 Strategic Grid)
    *   Represents a single hex. Buildings and points of interest are placed using Poisson-Disc sampling to ensure natural clustering.
*   **Tier 3: Local View** (100x100 Exploration Grid)
    *   A deterministic local grid featuring ambient nature (thickets, rock piles) based on the hex's biome.
*   **Tier 4: Tactical/Player View** (100x100 5ft-scale Grid)
    *   The most granular level, used for encounters and fine movement. It materializes building interiors and active NPCs/Enemies.

## 2. Frontend Rendering Architecture (PIXI.js)

The visual representation is handled entirely by PIXI.js within the VTT Client (`saga_vtt_client/src/components/MapRenderer.tsx`).

*   **Viewport Zoom Logic:** The frontend listens to the zoom level of the `pixi-viewport`. As the user zooms in, it automatically dispatches actions to change the `vttTier` (1 through 4) and fetches the corresponding map tier data from the backend.
*   **Tier 4 Rendering (`vttTier === 4`):**
    *   The frontend reads the `playerGrid` data from the `useWorldStore`.
    *   It iterates over a 100x100 array of strings (`encounter.grid`).
    *   **Walls (`WALL`):** Rendered as dark stone boxes (`0x44403c`).
    *   **Floors (`FLOOR`):** Rendered using a dirt texture (`dirt?.[0]`) tinted golden (`0x92400e`).
    *   **Debris (`DEBRIS`):** Rendered as grey squares (`0x78716c`).
    *   **Ambient Outdoors:** Assigned a pseudo-random grass texture based on the coordinate hash.
    *   **Tokens:** Players are drawn as blue circles (`0x3b82f6`), while NPCs are drawn as amber circles (`0xf59e0b`). Player visual assets (composite sprites) are passed in the data payload but currently rendered as simple shapes.

## 3. Backend Generation Logic & Data Sources (`saga_director`)

When the VTT requests Tier 4 data (via `/api/world/tactical/{hex_id}`), the `TacticalGenerator` (`saga_director/core/tactical_generator.py`) builds the environment deterministically.

*   **Deterministic Seeds:** The backend generates a reproducible random seed using the `hex_id` and the local coordinates (`lx`, `ly`) via the `WorldManager`. This ensures that revisiting the same coordinates yields the exact same layout.
*   **Building Generation (Box-and-Slice):** `BuildingGenerator.generate_layout` creates room bounds using recursive subdivision.
*   **Persistence Layer:** Before rendering walls, the engine checks a delta-state database (`wm.persistence.load_entity`) to see if the building is marked as "ruined." If ruined, walls are converted to `"DEBRIS"` or `"EMPTY"`. Otherwise, the edges of the rooms are marked as `"WALL"` with gaps for doors, and the interiors as `"FLOOR"`.

## 4. NPC Generation and Scheduling

NPCs are dynamically placed on the Tier 4 grid from two distinct sources:

### A. Scheduled Residents (Local Routine)
*   **Deterministic Locations:** The system assigns three random coordinates within the 100x100 grid to represent the NPC's `home`, `work`, and `tavern`.
*   **The `NPCScheduler`:** A daily routine is hardcoded based on the current in-game time (e.g., sleeping from 00:00 - 08:00, working from 09:00 - 17:00, at the tavern from 18:00 - 21:00).
*   **Linear Interpolation (LERP):** If the current hour falls during a "traveling" phase, the backend calculates the NPC's exact coordinate on the path between their origin and destination.
*   **Frontend Representation:** If an NPC's status is "traveling", the VTT draws a white stroke outline around their token to visually indicate movement.
*   **Mortality Check:** The persistence layer is checked. If an NPC's `is_dead` flag is true, they are not rendered.

### B. External NPCs (Quests & Events)
*   These are injected by the AI Narrator (`ContextAssembler`) based on active quests or local threat densities.
*   They are placed either at specific coordinates (`rx`, `ry`) or randomly if none are provided.
*   **Hostile/Friendly:** Their attitude dictates their color (Red for Hostile, Green for Friendly/Neutral).

## 5. Playability Audit & Recommendations

Based on the codebase analysis and the complex "Lead & Trail" skill system detailed in `crucial.md`, here is an assessment of the project's readiness for playtesting.

### Mechanics & Ruleset
*   **State:** Highly ambitious, featuring 36 tactical skills utilizing a Stamina/Focus drain mechanic and margin-of-victory combat resolution.
*   **Gap:** The VTT frontend currently lacks a comprehensive UI to support executing these specific skills natively. Complex interactions (e.g., "Advantage on damage vs Shields") require deep integration between the character, skill, and clash engines.
*   **Recommendation:** Begin with a "Vertical Slice" test. Implement a restricted subset of the 36 skills (e.g., one from each of the 12 command groups) to validate the core loop of stat selection and resource drain before implementing edge-case mechanics.

### Map Navigation
*   **State:** The 4-tier zoom system is functional.
*   **Gap:** The fluid mouse-wheel zooming can lead to accidental transitions out of the tactical view during fine movement or combat.
*   **Recommendation:** Implement a UI toggle to "lock" the zoom level or force players into Tier 4 during an encounter.

### Microservice Orchestration
*   **State:** The current architecture relies on 12 distributed microservices.
*   **Gap:** High potential for network latency and state desynchronization, especially when passing complex combat math across multiple Python/C++ boundaries and waiting on LLM narrative generation.
*   **Recommendation:** Strongly consider consolidating the 12 microservices into 4 core pillars (VTT, Rules Engine, Director, Architect) to drastically reduce network hops and simplify state management. In the interim, add robust loading indicators ("The Director is thinking...") to the VTT to handle latency gracefully.
